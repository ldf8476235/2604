# Boosting Module Implementation Plan
> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在现有 Delta Trade 项目中新增代肝服务模块，覆盖代肝服务大厅、代肝下单、代肝订单管理、进度查看、联系客服与售后申请，并保证 PC / H5 响应式体验可直接验收。

**Architecture:** 前端新增独立的代肝大厅、下单页与订单页，复用现有鉴权、OSS 上传与顶部/底部导航；后端新增 boosting 模块，使用独立服务表与订单表承载代肝业务，保持与账号交易订单解耦；支付先沿用测试环境即时成功链路，状态机与订单号规则与账号交易保持一致。

**Tech Stack:** React + TypeScript + Vite + React Router、Spring Boot + MyBatis-Plus + MySQL、Redis（沿用登录态）、OSS（沿用现有上传与预览能力）、SLF4J + MDC traceId（沿用现有日志体系）

---

### 1. 数据模型与后端表结构
- 新增 `boosting_service` 表，存储服务项配置、分类、价格、周期、保障说明、服务商、销量、状态、排序。
- 新增 `boosting_order` 表，存储代肝订单、账号信息、加密后的账号密码、特殊需求、支付/进度/售后状态、客服群号、时间节点。
- 新增 `boosting_progress_log` 表，存储代肝人员更新的进度记录。
- 启动期补表补列组件，兼容当前测试库。

### 2. 后端接口与业务服务
- 新增代肝服务大厅接口：
  - 分类/筛选元数据
  - 列表查询（类型、价格区间、周期、销量排序）
- 新增代肝下单接口：
  - 服务详情
  - 创建订单
  - 测试环境支付确认
- 新增代肝订单管理接口：
  - 订单列表（状态、时间范围）
  - 订单详情
  - 进度列表
  - 售后申请
- 账号密码采用项目内服务做加密存储，并提供脱敏/有限展示字段。
- 日志沿用现有 SLF4J + MDC traceId，覆盖查询、创建、支付、进度、售后关键节点。

### 3. 前端页面与交互
- 新增 `/boosting` 代肝服务大厅：
  - 服务分类、筛选、排序、服务卡片
  - PC/H5 双布局
- 新增 `/boosting/:serviceNo/order` 下单页：
  - 服务信息区
  - 账号信息表单
  - 协议勾选
  - 提交下单与测试支付按钮
- 新增 `/boosting/orders` 代肝订单页：
  - 订单角色以“我的代肝订单”为单一视角
  - 状态筛选、时间筛选、列表、详情弹层
  - 进度时间线、联系客服入口、售后申请
- 顶部导航、H5 快捷入口和底部导航增加代肝入口。

### 4. 联调与验收
- 生成测试服务数据与测试订单数据，保证页面打开即可验收。
- 验证：
  - 后端编译通过
  - 前端构建通过
  - PC/H5 页面无横向溢出
  - 下单、支付、订单列表、详情、进度、售后全链路可用

### 5. 风险与取舍
- 本轮不接真实支付与真实 IM，仅保留可替换入口和测试支付成功链路。
- 本轮不开放代肝人员后台，只在数据库中准备进度日志模型，并提供前台只读能力。
- 若后续需要工作室承接代肝，可在 `boosting_service` 上增加 provider 维度扩展，不破坏本轮表结构。
